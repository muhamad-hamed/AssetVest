# ADR-0005 — Authentication: Self-Issued JWT

**Status:** Accepted  
**Date:** 2026-05-30

## Context

AssetVest needs authentication for its single user. Options: external IdP (Auth0, Azure AD B2C, Keycloak), or self-issued JWT tokens generated by the API itself.

## Decision

Use **self-issued JWT tokens** generated by `AssetVest.Api`.

- Access token: short-lived (15 minutes)
- Refresh token: long-lived (7 days), stored hashed in database
- Algorithm: HS256 (symmetric, single service)
- Library: `System.IdentityModel.Tokens.Jwt` / `Microsoft.AspNetCore.Authentication.JwtBearer`

## Token Claims

```json
{
  "sub": "user-guid",
  "email": "user@example.com",
  "jti": "unique-token-id",
  "iat": 1234567890,
  "exp": 1234568790
}
```

## Refresh Token Strategy

- Stored in `RefreshToken` table (hashed, not plain)
- Rotation on each use (old token revoked, new issued)
- `RevokedAt` set on logout or suspicious activity
- Max active refresh tokens per user: 5 (device management)

## Security Notes

- JWT signing key stored in environment variable / user secrets — never in appsettings.json
- HTTPS enforced in production
- Refresh token transmitted via HttpOnly cookie (recommended) or response body
- Failed login attempts tracked (future: rate limiting / lockout)

## Consequences

- No external IdP dependency
- Full control over token lifecycle
- Must implement token refresh endpoint
- Must handle token revocation via database lookup on refresh
